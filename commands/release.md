Выполни сохранение стабильной версии проекта. Аргументы: $ARGUMENTS

## Шаги

### 0. Определи проект
Определи проект по текущей рабочей директории: найди ближайший каталог с `package.json` и `lib/version.ts`. Используй имя из `package.json` → `name` как название проекта.

Если проект указан в аргументах — перейди в его папку. Если в текущей директории нет `lib/version.ts` — спроси пользователя, в какой папке проект.

Далее все пути относительно папки проекта.

### 1. Определи новую версию
Если версия передана в аргументах — используй её. Иначе прочитай текущую версию из `lib/version.ts` проекта и предложи следующую patch-версию (например, 0.6.0 → 0.6.1).

### 2. Собери список изменений
Посмотри на `git diff HEAD` и недавние коммиты (`git log --oneline -20`) в папке проекта, чтобы понять, что изменилось с последней версии. Сформируй краткий список изменений на русском языке для changelog.

**Стиль описания:**
- Пиши с точки зрения пользователя: **что он теперь может делать**, а не как это реализовано внутри
- Избегай низкоуровневых деталей: названия эндпоинтов (`GET /api/...`), параметры URL (`?presenter=`), названия библиотек (`Prisma`, `Zod`). Вместо этого — описание результата на понятном языке
- Технологические доработки описывай, если они значимы (новая интеграция, улучшение производительности, безопасность), но формулируй по сути, а не по реализации
- Категории: `[Новое]`, `[Улучшение]`, `[Исправление]`

### 3. Покажи changelog пользователю для подтверждения
Выведи собранный список изменений пользователю и спроси, всё ли корректно. Если пользователь попросит что-то изменить, добавить или убрать — доработай changelog по его замечаниям. Переходи к следующему шагу только после явного подтверждения.

### 4. Обнови `lib/version.ts`
- Замени значение `APP_VERSION` на новую версию
- Добавь новый объект в начало массива `CHANGELOG` с полями:
  - `version` — новая версия
  - `date` — сегодняшняя дата в формате `YYYY-MM-DD`
  - `changes` — массив строк с описанием изменений. Формат: `"[Категория] Описание"`. Категории: `[Новое]`, `[Улучшение]`, `[Исправление]`. Описание бизнес-ориентированное (см. шаг 2)

### 5. Обнови `CLAUDE.md`
Замени строку `Версия X.X.X` в заголовке «О проекте» на новую версию.

### 6. Обнови `data/roadmap.json` (если файл существует)
1. Обнови `meta.version` и `meta.lastUpdated` на новую версию и сегодняшнюю дату
2. Проверь, есть ли задачи со статусом `OPEN`, которые явно были выполнены в этом релизе (по списку изменений из шага 2). Если есть — обнови их: `status` → `"DONE"`, добавь `version` с номером текущего релиза
3. Если в этом релизе реализованы задачи, которых **нет в массиве `tasks`** — добавь их (id, title, description, type, priority, status, version, wave, designDoc)
4. Запусти `npx tsx scripts/check-roadmap-sync.ts` для валидации

### 6.1. Обнови пользовательскую документацию (если файл `lib/docs-content.ts` существует)
Прочитай `lib/docs-content.ts` и проверь, актуально ли содержимое с учётом изменений в этом релизе. Если в релизе добавлены новые разделы, страницы, функции или существенно изменена логика существующих — обнови соответствующие секции в документации. Добавь описание новых возможностей, обнови существующие разделы при необходимости. Не трогай разделы, которые не затронуты изменениями.

Дополнительно проверь связанные файлы:
- `lib/docs-context.ts` — если добавлена новая страница (новый `app/*/page.tsx`), добавь маппинг pathname → section id в `PATHNAME_TO_SECTION`
- `DOC_NAV_GROUPS` в `lib/docs-content.ts` — если добавлена новая секция документации, включи её id в соответствующую навигационную группу
- `components/docs/DocsPage.tsx` — если новая группа использует иконку, которой нет в `NAV_ICONS`, добавь импорт из `lucide-react` и запись в маппинг

### 6.2. Обнови техническую документацию (если каталог `data/` содержит `*-design.md` или `*-architecture.md`)
Проверь файлы `data/*-design.md` и `data/*-architecture.md`. Если изменения в релизе затрагивают описанную в них архитектуру (новые модели, API-маршруты, паттерны авторизации, расчётные формулы) — обнови соответствующие разделы.

### 6.3. Синхронизируй `docs/design/roadmap.md` с `data/roadmap.json`
Запусти `npx tsx scripts/generate-roadmap-md.ts` — скрипт автоматически перегенерирует `docs/design/roadmap.md` из JSON (заголовок, статус, таблицы волн, сводку, открытые задачи, дизайн-документы).

### 7. Проверь Prisma-миграции
Убедись, что все поля/таблицы/enum'ы из `prisma/schema.prisma` покрыты миграциями в `prisma/migrations/`.

**Основной способ:** `prisma migrate diff --from-config-datasource --to-schema prisma/schema.prisma --script`. Если diff не пустой — создай миграцию.

**Fallback (если команда упала из-за shadow DB или других ошибок):** извлеки из `schema.prisma` все имена полей (колонок) каждой модели и проверь через grep, что каждое поле присутствует хотя бы в одном файле `migration.sql`. Особое внимание — полям, добавленным в текущем релизе (они видны в `git diff`). Для каждого пропущенного поля создай миграцию.

**Формат миграции:** создай папку `prisma/migrations/{timestamp}_{name}/migration.sql`. SQL должен быть идемпотентным (`IF NOT EXISTS`, `DO $$ ... EXCEPTION WHEN duplicate_object`).

### 8. Проверь TypeScript
Запусти `npx tsc --noEmit` в папке проекта. Если есть ошибки типов — исправь их перед коммитом.

### 8.1. Запусти тесты
Запусти `npm run test:unit` в папке проекта. Если тесты падают — исправь причину перед коммитом.

**E2E тесты обязательны при релизе.** Если в проекте есть `npm run test:e2e:auto`, запускай именно его: команда должна сама выполнить release smoke и добавить full/specialized E2E-профили по policy проекта. Если `test:e2e:auto` нет, но есть `npm run check:e2e-policy`, сначала запусти policy и следуй его решению без вопроса пользователю. Минимальный обязательный gate — `npm run test:e2e` как release smoke. Если policy выводит `full_required=true`, дополнительно запусти `npm run test:e2e:full`; full regression считается покрывающим specialized Playwright spec-файлы, если проект не документирует обратное. Если `full_required=false`, но policy перечисляет `a11y`, `responsive` или `visual`, запусти соответствующие `npm run test:e2e:*` команды. Если E2E падают — исправь перед коммитом.

### 9. Синхронизируй с remote
Выполни `npm run sync` из папки проекта, чтобы подтянуть возможные изменения от других разработчиков.

### 10. Создай git-коммит
Добавь **все** изменённые и новые файлы в папке проекта (`git add -A`), включая незакоммиченные изменения, которые были до начала релиза. Создай коммит с заголовком и телом:
```
release: v{новая_версия}

- пункт changelog 1
- пункт changelog 2
- ...
```
В теле коммита перечисли все пункты из подтверждённого changelog (по одному на строку, с дефисом). **НЕ добавляй `[deploy]`** в этот коммит — решение о деплое принимается после успешного пуша.

### 11. Запушь
Выполни `git push` из папки проекта. Если pre-push хуки обнаружат ошибки (lint, тесты) — исправь их отдельными коммитами и повтори push.

### 12. Определи режим деплоя проекта

Прежде чем что-то спрашивать про деплой, **проверь как настроен Vercel** в этом проекте:

1. Прочитай `vercel.json` (если есть) или поищи `git.deploymentEnabled` через `vercel project inspect <name>` если возможно
2. Альтернатива: посмотри последние коммиты `git log --oneline -10` — если все недавние пуши без `[deploy]` маркера приводили к деплою (об этом обычно знает пользователь), то auto-deploy включен
3. Если в `vercel.json` есть `"git": { "deploymentEnabled": false }` или подобное — деплой ручной (нужен `[deploy]` маркер)
4. Если нет настроек или `deploymentEnabled: true` — деплой автоматический

**Спроси пользователя один раз для нового проекта**: «У этого проекта auto-deploy на push, или нужен ручной маркер `[deploy]`?» — и **запомни ответ в memory** через `feedback_<project>_deploy.md`, чтобы в следующих релизах не спрашивать.

### 13. Действия по результату

**Если auto-deploy:**
- Vercel уже запустит сборку после `git push` из шага 11
- Сообщи пользователю: «Vercel запустит автоматический деплой v{version}»
- НЕ создавай дополнительный коммит с `[deploy]`

**Если ручной деплой:**
- Спроси: «Задеплоить на Vercel?» (да/нет)
- Если **да** — `git commit --allow-empty -m "chore: trigger deploy [deploy]"` и `git push`. Vercel проверяет только HEAD-коммит, поэтому маркер должен быть именно в последнем коммите.
- Если **нет** — ничего не делать.

Выведи итоговое сообщение: проект, версия, дата, список изменений из changelog. Укажи статус деплоя (автоматический/ручной/пропущен).
