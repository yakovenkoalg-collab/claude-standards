Выполни сохранение стабильной версии проекта. Аргументы: $ARGUMENTS

## Шаги

### 0. Определи проект
Определи проект по текущей рабочей директории: найди ближайший каталог с `package.json` и `lib/version.ts`. Используй имя из `package.json` → `name` как название проекта.

Если проект указан в аргументах — перейди в его папку. Если в текущей директории нет `lib/version.ts` — спроси пользователя, в какой папке проект.

Далее все пути относительно папки проекта.

### 1. Зафиксируй source и scope релиза ДО версии и changelog
Сначала проверь, откуда именно будет выпускаться релиз. Не строй changelog от активной рабочей директории без проверки ветки, upstream и локальных изменений.

Выполни и кратко покажи пользователю:
- `git status -sb`
- текущую ветку
- upstream-ветку, если она настроена
- default remote branch через `origin/HEAD`, если доступно
- коммиты, которые есть локально сверх release base: `git log --oneline <base>..HEAD`
- коммиты, которых не хватает локально: `git log --oneline HEAD..<base>`
- committed diff относительно release base: `git diff --name-status <base>...HEAD`
- незакоммиченные tracked-изменения: `git diff --name-status HEAD`
- untracked non-ignored файлы: `git ls-files --others --exclude-standard`

Правила выбора scope:
- По умолчанию release source — default branch (`main`/`master` через `origin/HEAD`) или явно переданная пользователем ветка/ref.
- Не выпускай релиз молча с feature-ветки. Если текущая ветка не является выбранной release branch — остановись и спроси, что делать: переключиться на default branch, релизить текущую ветку, либо сначала merge/cherry-pick конкретных коммитов.
- Если локальная release branch ahead/behind относительно upstream — покажи точные коммиты и попроси подтвердить scope до составления changelog.
- Если есть незакоммиченные или untracked файлы — классифицируй их как release changes, local work или generated artifacts. Не включай untracked/generated файлы в релиз без явного подтверждения пользователя.
- Используй подтверждённые release base и scope во всех следующих шагах: changelog, проверки, staging, commit.

### 2. Определи новую версию
Если версия передана в аргументах — используй её. Иначе прочитай текущую версию из `lib/version.ts` проекта и предложи следующую patch-версию (например, 0.6.0 → 0.6.1).

### 3. Собери список изменений
Собирай changelog только из подтверждённого scope из шага 1:
- committed changes: `git log --oneline <release-base>..HEAD` и `git diff --stat <release-base>...HEAD`
- approved uncommitted tracked changes: `git diff HEAD`
- approved untracked files: читай только явно подтверждённые пути

Не используй `git log --oneline -20` как источник scope релиза; это только контекст, если нужен обзор истории.

Сформируй краткий список изменений на русском языке для changelog.

**Стиль описания:**
- Пиши с точки зрения пользователя: **что он теперь может делать**, а не как это реализовано внутри
- Избегай низкоуровневых деталей: названия эндпоинтов (`GET /api/...`), параметры URL (`?presenter=`), названия библиотек (`Prisma`, `Zod`). Вместо этого — описание результата на понятном языке
- Технологические доработки описывай, если они значимы (новая интеграция, улучшение производительности, безопасность), но формулируй по сути, а не по реализации
- Категории: `[Новое]`, `[Улучшение]`, `[Исправление]`

### 4. Покажи scope, версию и changelog пользователю для подтверждения
Покажи:
- release source branch/ref
- release base
- included commits
- included uncommitted/untracked paths
- предлагаемую версию
- changelog

Спроси, всё ли корректно. Если пользователь попросит что-то изменить, добавить или убрать — доработай scope/changelog по его замечаниям. Переходи к следующему шагу только после явного подтверждения.

### 5. Обнови `lib/version.ts`
- Замени значение `APP_VERSION` на новую версию
- Добавь новый объект в начало массива `CHANGELOG` с полями:
  - `version` — новая версия
  - `date` — сегодняшняя дата в формате `YYYY-MM-DD`
  - `changes` — массив строк с описанием изменений. Формат: `"[Категория] Описание"`. Категории: `[Новое]`, `[Улучшение]`, `[Исправление]`. Описание бизнес-ориентированное (см. шаг 3)

### 6. Обнови `CLAUDE.md`
Замени строку `Версия X.X.X` в заголовке «О проекте» на новую версию.

### 7. Обнови `data/roadmap.json` (если файл существует)
1. Обнови `meta.version` и `meta.lastUpdated` на новую версию и сегодняшнюю дату
2. Проверь, есть ли задачи со статусом `OPEN`, которые явно были выполнены в этом релизе (по списку изменений из шага 3). Если есть — обнови их: `status` → `"DONE"`, добавь `version` с номером текущего релиза
3. Если в этом релизе реализованы задачи, которых **нет в массиве `tasks`** — добавь их (id, title, description, type, priority, status, version, wave, designDoc)
4. Запусти `npx tsx scripts/check-roadmap-sync.ts` для валидации, если скрипт существует

### 7.1. Обнови пользовательскую документацию (если файл `lib/docs-content.ts` существует)
Прочитай `lib/docs-content.ts` и проверь, актуально ли содержимое с учётом изменений в этом релизе. Если в релизе добавлены новые разделы, страницы, функции или существенно изменена логика существующих — обнови соответствующие секции в документации. Добавь описание новых возможностей, обнови существующие разделы при необходимости. Не трогай разделы, которые не затронуты изменениями.

Дополнительно проверь связанные файлы:
- `lib/docs-context.ts` — если добавлена новая страница (новый `app/*/page.tsx`), добавь маппинг pathname → section id в `PATHNAME_TO_SECTION`
- `DOC_NAV_GROUPS` в `lib/docs-content.ts` — если добавлена новая секция документации, включи её id в соответствующую навигационную группу
- `components/docs/DocsPage.tsx` — если новая группа использует иконку, которой нет в `NAV_ICONS`, добавь импорт из `lucide-react` и запись в маппинг

### 7.2. Обнови техническую документацию (если каталог `data/` содержит `*-design.md` или `*-architecture.md`)
Проверь файлы `data/*-design.md` и `data/*-architecture.md`. Если изменения в релизе затрагивают описанную в них архитектуру (новые модели, API-маршруты, паттерны авторизации, расчётные формулы) — обнови соответствующие разделы.

### 7.3. Синхронизируй `docs/design/roadmap.md` с `data/roadmap.json`
Если `scripts/generate-roadmap-md.ts` существует, запусти `npx tsx scripts/generate-roadmap-md.ts` — скрипт автоматически перегенерирует `docs/design/roadmap.md` из JSON (заголовок, статус, таблицы волн, сводку, открытые задачи, дизайн-документы).

### 8. Проверь Prisma-миграции
Убедись, что все поля/таблицы/enum'ы из `prisma/schema.prisma` покрыты миграциями в `prisma/migrations/`.

**Основной способ:** `prisma migrate diff --from-config-datasource --to-schema prisma/schema.prisma --script`. Если diff не пустой — создай миграцию.

**Fallback (если команда упала из-за shadow DB или других ошибок):** проверь поля/таблицы/enum'ы, изменённые в подтверждённом release scope, и убедись, что они присутствуют в миграциях. Для каждого пропущенного изменения создай миграцию.

**Формат миграции:** создай папку `prisma/migrations/{timestamp}_{name}/migration.sql`. SQL должен быть идемпотентным (`IF NOT EXISTS`, `DO $$ ... EXCEPTION WHEN duplicate_object`).

### 9. Проверь TypeScript
Запусти `npx tsc --noEmit` в папке проекта. Если есть ошибки типов — исправь их перед коммитом.

### 9.1. Запусти тесты
Запусти `npm run test:unit` в папке проекта. Если тесты падают — исправь причину перед коммитом.

**E2E тесты обязательны при релизе.** Если в проекте есть `npm run test:e2e:auto`, запускай именно его: команда должна сама выполнить release smoke и добавить full/specialized E2E-профили по policy проекта. Если `test:e2e:auto` нет, но есть `npm run check:e2e-policy`, сначала запусти policy и следуй его решению без вопроса пользователю. Минимальный обязательный gate — `npm run test:e2e` как release smoke. Если policy выводит `full_required=true`, дополнительно запусти `npm run test:e2e:full`; full regression считается покрывающим specialized Playwright spec-файлы, если проект не документирует обратное. Если `full_required=false`, но policy перечисляет `a11y`, `responsive` или `visual`, запусти соответствующие `npm run test:e2e:*` команды. Если E2E падают — исправь перед коммитом.

### 10. Синхронизируй с remote
Выполни `npm run sync` из папки проекта, если скрипт существует, чтобы подтянуть возможные изменения от других разработчиков.

После sync/rebase повтори scope-проверку из шага 1. Если появились новые коммиты, конфликты или изменился diff релиза — покажи обновлённый scope и влияние на changelog до commit.

### 11. Создай git-коммит
Добавь в staging только подтверждённый release scope плюс release-файлы (`lib/version.ts`, `CLAUDE.md`, roadmap/docs/migrations, если они реально обновлены).

Не выполняй слепой `git add -A` и не добавляй unrelated local work или generated artifacts без явного подтверждения.

Создай коммит с заголовком и телом:
```
release: v{новая_версия}

- пункт changelog 1
- пункт changelog 2
- ...
```
В теле коммита перечисли все пункты из подтверждённого changelog (по одному на строку, с дефисом). **НЕ добавляй `[deploy]`** в этот коммит — решение о деплое принимается после успешного пуша.

### 12. Запушь
Если в `package.json` есть скрипт `push:checked`, выполни `npm run push:checked` из папки проекта. Он сначала запускает pre-push проверки локально, до открытия SSH-соединения, а затем делает короткий `git push --no-verify`.

Если скрипта нет — выполни `git push`. Если pre-push хуки обнаружат ошибки (lint, тесты) — исправь их отдельными коммитами и повтори push.

### 13. Определи режим деплоя проекта

Прежде чем что-то спрашивать про деплой, проверь как настроен Vercel в этом проекте:

1. Прочитай `vercel.json` и проверь `git.deploymentEnabled`, если файл существует
2. Если возможно, проверь настройки проекта через локальный Vercel CLI
3. Если `vercel.json` или Vercel CLI подтверждает auto-deploy — считай деплой автоматическим
4. Если `vercel.json` или Vercel CLI подтверждает `deploymentEnabled: false` — деплой ручной (нужен `[deploy]` marker)

Не делай вывод об auto-deploy только по README/CLAUDE/docs или по недавним коммитам. Если `vercel.json` отсутствует и Vercel CLI недоступен, спроси пользователя в конце релиза, нужен ли отдельный deploy commit с `[deploy]` marker.

### 14. Действия по результату

**Если auto-deploy:**
- Vercel запустит сборку после `git push` из шага 12
- Сообщи пользователю: «Vercel запустит автоматический деплой v{version}»
- НЕ создавай дополнительный коммит с `[deploy]`

**Если ручной деплой:**
- Спроси: «Задеплоить на Vercel?» (да/нет)
- Если **да** — `git commit --allow-empty -m "chore: trigger deploy [deploy]"` и `npm run push:checked` (или `git push`, если скрипта нет). Vercel проверяет только HEAD-коммит, поэтому маркер должен быть именно в последнем коммите.
- Если **нет** — ничего не делать.

### 15. Сообщи результат
Выведи итоговое сообщение: проект, версия, дата, список изменений из changelog. Укажи статус push и деплоя (автоматический/ручной/пропущен), а также любые пропущенные проверки или остаточные риски.
